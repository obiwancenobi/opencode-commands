Project Overview

CommandKenobi is a documentation-only repository that provides slash command specifications for git operations and planning workflows. The core idea: write once, run across three AI coding assistants.

There is no executable code here. Instead, each file is a structured Markdown specification that an AI assistant reads and follows. When you type /commit-all in your AI tool, it loads the corresponding spec file and executes the workflow described within it.

Language & Framework

Primary language: Markdown
Framework: None (pure documentation for AI assistants)
Repository type: Static command specifications

Command Discovery and Auto-Loading

When a user clones CommandKenobi into their project, the AI tool automatically discovers command files from its designated directory. Claude Code scans .claude/commands/, OpenCode scans .opencode/commands/, and Kilo Code scans .kilocode/workflows/.

No configuration is required. The directories follow each tool's conventions, so commands appear as slash commands automatically. The user types something like /commit-all in their AI assistant conversation, and the tool loads the corresponding Markdown spec.

Directory Layout

Each tool has its own directory with the same 9 command files. This multi-format approach means teams using different AI assistants can share the same repository.

Context Injection via CLAUDE.md and AGENTS.md

Before executing any command, the AI assistant loads repository-level instructions. Claude Code reads CLAUDE.md automatically as project instructions. This file establishes the repository overview, architecture, command file structure, git conventions, and multi-tool interaction patterns.

AGENTS.md provides comprehensive agent guidelines including code style, naming conventions, error handling patterns, and safety rules. Together, these two files set the "rules of engagement" that govern all subsequent command execution.

Step 4 of 9

Frontmatter Parsing and Permission Scoping

When a command is invoked, the AI assistant reads the corresponding Markdown file and parses its YAML frontmatter. The frontmatter acts as a security contract: allowed-tools restricts which tools the agent can use during execution.

For example, /commit-all only permits Bash(git:*), Read, Grep, Glob. It cannot write files, launch subagents, or use the GitHub CLI. The more complex /create-pr command has a wider permission scope including Bash(gh:*), Bash(brew:*), and the Task tool for subagent capabilities.

.claude/commands/commit-all.md:1-4
 1---
 2description: Create a conventional commit from all changes with user approval
 3allowed-tools: Bash(git:*), Read, Grep, Glob
 4---

Compare: Three Formats

Each AI tool format uses different frontmatter. The content below each command is the same workflow — only the metadata and interaction primitives differ.

.opencode/commands/commit-all.md:1-4 (OpenCode)
 1---
 2description: Create a conventional commit from all changes with user approval
 3agent: build
 4---
.kilocode/workflows/commit-all.md:1-3 (Kilo Code)
 1# Commit All
 2
 3Create a conventional commit from all changes with user approval.

Key Takeaways

  • Frontmatter defines the permission scope for each command
  • allowed-tools (Claude Code) vs agent (OpenCode) vs no frontmatter (Kilo Code)
  • Simpler commands get tighter permissions; complex commands get wider scopes
Step 5 of 9

Pre-flight Checks — Validation Before Any Action

Every command starts with safety validations. The agent verifies git configuration (user.name, user.email), checks the current branch, and detects protected branches (main, master, develop, release/*).

This "fail fast" philosophy is a core design decision. No destructive operation proceeds without passing all checks. If the user is on a protected branch, they get three options: commit anyway, create a new branch, or cancel.

.claude/commands/commit-all.md:8-25

11   - Run: Bash(git config user.name)
12     If empty: "Git user not configured. Set git config user.name first."
13   - Run: Bash(git config user.email)
14     If empty: "Git user not configured. Set git config user.email first."
15
162. **Check Current Branch**
17   - Run: Bash(git branch --show-current)
18   If branch is `main`, `master`, `develop`, or matches `release/*`,
19   warn user:
20
21   You are on a protected branch. What would you like to do?
22
23   1. Commit Anyway — Commit directly to protected branch
24   2. Create Branch — Create a new branch and commit there
25   3. Cancel — Abort the commit

Step 6 of 9

Parallel Change Analysis

The agent runs multiple git commands in parallel to gather context. This is a deliberate performance pattern: independent reads can happen simultaneously rather than sequentially.

The agent examines which files are staged versus unstaged, the full diff of all changes, and recent commit patterns for context. For the /create-pr command, this phase also gathers PR-specific context including all commits since diverging from the base branch.

.claude/commands/commit-all.md:38-51
38### 1. Read All Changes
39Run in parallel:
40- Run: Bash(git status)
41- Run: Bash(git diff --cached --stat)
42- Run: Bash(git diff --cached)
43- Run: Bash(git diff --stat)
44- Run: Bash(git diff)
45- Run: Bash(git log --oneline -5)
46
47### 2. Analyze Changes
48Examine:
49- Which files are staged vs unstaged
50- Full diff of all changes
51- Recent commit patterns for context

This data feeds the next step. The agent needs to understand the nature of changes — are they adding a feature, fixing a bug, updating docs? — to generate an appropriate commit message. The recent commit log provides pattern context so the generated message matches the project's existing style.

Key Takeaways

  • Six git commands run in parallel for performance
  • Both staged and unstaged changes are analyzed
  • Recent commit history informs message style
Step 7 of 9

Output Generation

The agent generates output based on the analyzed changes. For git commands, this means a conventional commit message. For content commands like /generate-social-content, this involves launching parallel subagents that each produce platform-specific content.

The conventional commit format is strictly enforced: type(scope): subject. Types are limited to: feat, fix, docs, style, refactor, test, chore, perf. The subject must be under 72 characters, use imperative mood, and have no trailing period.

.claude/commands/commit-all.md:53-58
53### 3. Generate Commit Message
54- Format: type(scope): subject
55- Types: feat, fix, docs, style, refactor, test, chore, perf
56- Scope: primary feature, directory, or component
57- Subject: under 72 characters, imperative mood, no period
58- Focus on the most significant change

Subagent Pattern (Content Commands)

Commands like /generate-social-content use a fan-out/fan-in pattern. The orchestrator gathers context, then launches one subagent per platform simultaneously. Each subagent writes to a separate file. After all complete, a humanizer subagent post-processes the output to remove AI writing patterns.

Step 8 of 9

User Approval Gate

No destructive operation executes without explicit user confirmation. The agent presents the generated output and asks the user to choose from numbered options. This is the critical safety gate across all commands.

If the user picks "Suggest Again," the flow loops back to the generation step. The approval pattern is consistent, but the interaction mechanism differs per tool: Claude Code uses inline chat prompts, OpenCode uses the question tool with structured YAML, and Kilo Code uses ask_user.

Step 9 of 9

Execution and Error Handling

On user acceptance, the agent executes the operation. For /commit-all, this means staging all files and committing. For /push-all, an additional push to remote follows. For /create-pr, execution spans two full phases: first commit and push, then PR creation.

Every error path has a specific, actionable message. The /create-pr command demonstrates the most sophisticated fallback chain in the repository: try gh CLI, offer to install via brew, offer to authenticate, and finally fall back to opening a browser compare URL.

.claude/commands/commit-all.md:72-82
72### 5. Handle Response
73- If response = "Accept":
74    Run Bash(git add . && git commit -m "YOUR_COMMIT_MESSAGE")
75- If response = "Suggest Again": Generate new message, present again
76- If response = "Cancel": Abort with "Commit cancelled."
77

Error Handling Pattern

Every command file ends with an ## Error Handling section that enumerates specific failure modes with exact messages and recovery actions. The pattern is consistent: detect, inform with a specific message, offer recovery options.

Key Takeaways

  • Execution only happens after explicit user approval
  • Every error path provides specific, actionable recovery steps
  • Complex commands use multi-level fallback chains for graceful degradation
